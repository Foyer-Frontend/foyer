   They depend on legacy `views.hpp` (`State`, `View` enum). Plan:

   to drain the chain-launch. Confirmed working for libretro players;
   not yet validated for Switch native title chain (see #2).

## Repo layout — what's compiled vs on-disk-only in 0.6.0

Files **on disk** but **NOT compiled** in the brls build (kept for
cherry-picking into Phase G cleanup):

- `browser/src/views.cpp` — the full legacy nanovg renderer + view
  state machine
- `browser/src/theme.cpp` — legacy JSONC theme loader (replaced by
  brls's built-in theme system)
- `browser/src/boot_splash.cpp` — legacy splash; brls doesn't expose
  a hookable splash, we'll roll our own activity if needed
- `browser/src/session.cpp` + `views.hpp` — view-state stash for
  resume-from-player. Needs the header refactored before it lands.
- `browser/src/switch_titles.cpp` — NACP icon decoder + chain-launcher
- `browser/src/mtp.cpp` — libhaze MTP server (USB rom drop-off)
- `browser/src/seed_assets.cpp` — bezel/cheat seed paths

Files compiled into the **brls build** (`browser/src/*` minus the
above):
- `main.cpp`, `home_activity.cpp`, `system_activity.cpp`,
  `game_activity.cpp`, `search_activity.cpp`, `settings_activity.cpp`,
  `settings_tab.cpp`, `power_activity.cpp`, `wizard_activity.cpp`
- `power_actions.cpp`, `library_state.cpp`, `manifest_cache.cpp`,
  `first_run.cpp`, `self_update.cpp`, `hos_status.cpp`,
  `launch.cpp` (with the `switch://` branch gated off)

Plus all of `foyer_shared` (services), borealis library (FetchContent),
`libhaze` (still pulled but not yet linked into the brls path).

## Build invocation (unchanged across the desktop move)

```sh
export DEVKITPRO=/opt/devkitpro
cd foyer-workspace/foyer
cmake --preset Release
cmake --build build/Release -j$(nproc)
# → build/Release/foyer.nro
```

First configure pulls XITRIX/borealis (~50 MB clone) + brls's vendored
fmt / yoga / tweeny / tinyxml2 / libretro-common subset. Subsequent
configures are fast.

## Open questions for the new desktop

- Is the Switch + dock readily testable from the desktop? Hardware
  iteration is the actual bottleneck — without a flash-and-test loop
  the wizard's network/install code can't be validated.
- Any preference for nxlink? `log::init_nxlink()` is wired and dumps
  the same lines that go to the per-run log file. If the desktop
  has nxlink set up, that's faster than copying log files off the
  SD.
