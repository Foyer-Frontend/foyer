## Foyer feature surface (what actually works on device)

- **Browser**: ES-DE-style carousel + game list + sidebar with scraped
  metadata + cover.
- **Input**: D-pad + L/ZL/R/ZR shoulder navigation with hold-to-spin
  auto-repeat; touch on Home/System/Settings/popup/quit-confirm/update-confirm.
- **Per-game launcher**: chains into `foyer-<core>.nro` players.
- **Pause overlay** (player-side): save states, aspect ratio, libretro
  options, Quit. **Touch on the pause overlay is NOT yet wired** — only
  D-pad + buttons.
- **Self-update**: boot-time check fetches `foyer-manifest.json`,
  yes/no modal, atomic rename of `foyer.nro.new` on next boot.
- **Cores install/update**: Settings → Emulator catalogues every
  manifest entry (install action for not-installed). Settings →
  Updates lists ONLY installed cores with size mismatch (pending
  updates) + foyer self-update at the top.
- **Per-system default core**: Settings → Emulator → "Default core per
  system" cycle picker for every system in `system_db`.
- **System art**: 425 PNGs (logo + splash for every system Art Book Next
  ES-DE covers) bundled in `assets/romfs/systems/`. License terms in
  `assets/LICENSE-ART.md` (CC-BY-NC-SA 2.0, attribution to
  anthonycaccese, ShareAlike for any redistribution of that directory).

## Decisions parked (need user input next session)

### gpsp — drop or invest?

gpsp's arm64 dynarec is incomplete upstream — only `arm/arm_stub.S`
(32-bit) defines `ram_translation_cache` / `rom_translation_cache`.
`arm/arm64_stub.S` exists but lacks both the cache buffers and the
instruction emitter. Completing it = weeks of systems work.
Without dynarec, gpsp's interpreter is slower than mGBA's. **Recommend
dropping gpsp**; mGBA + vba_next cover GBA.

### Adding heavy cores

## Recurring CI hazards (carry-over knowledge)

- **`pkg.devkitpro.org` 403s** on `dkp-pacman` db sync. The
  `foyer-cores-builder` GHCR image bakes portlibs in once to dodge
  this for the cores matrix. **`foyer/.github/workflows/release.yml`
  doesn't use that image** — cross-repo GHCR auth from the foyer repo
  was unreliable, so foyer's release uses runtime pacman with a
  10-attempt retry loop. v0.2.8 needed one rerun (CDN flake) before
  succeeding.
- **GHCR cross-repo auth**: `foyer-frontend/foyer-cores`-published
  packages aren't auto-pullable from `foyer-frontend/foyer`'s workflow
  even with `packages: read` permission. The "Manage Actions access"
  toggle on the package settings page is the workaround but didn't
  propagate cleanly during this session. If you ever switch foyer's
  release to use the GHCR image, double-check the package permits the
  foyer repo before tagging.
- **`gh run rerun --failed` re-uses the trigger ref's code**, not
  current main. To get a recipe fix into a tagged release, tag a new
  version — don't rerun an old failed job and expect new code.
- **CMake's `target_compile_options` collapses consecutive `-include
  X / -include Y` pairs**. Use the `SHELL:` generator-expression
  prefix to keep them paired. See `bsnes_hd_beta.cmake` for the
  pattern.
- **`xxd -i -n` isn't on every container's xxd** (the `-n` flag was
  added in vim 8.0). For binary→C codegen prefer the pure-CMake
  `recipes/bin2c.cmake` script — see `sameboy.cmake` for the
  add_custom_command pattern.

## Build verification cheatsheet

```sh
# devkitPro env
source /etc/profile.d/devkit-env.sh

# Foyer browser (fast, ~1 min full + linktime)
cmake --build /home/tobagin/foyer-workspace/foyer/build/Release -j$(nproc)

# One core, fresh dir
cmake -B /home/tobagin/foyer-workspace/foyer/build/Player-<core> \
      -S /home/tobagin/foyer-workspace/foyer \
      --toolchain $DEVKITPRO/cmake/Switch.cmake \
      -DCMAKE_BUILD_TYPE=MinSizeRel \
      -DBUILD_BROWSER=OFF -DBUILD_PLAYER=ON \
      -DFOYER_CORE=<core>
cmake --build /home/tobagin/foyer-workspace/foyer/build/Player-<core> -j$(nproc)
```
